# Kubernetes Secrets Management Guide\n\nThis guide explains how to properly manage sensitive data in your Kubernetes deployments.\n\n## ⚠️ CRITICAL: Before Deploying to Production\n\n### 1. Remove All Hardcoded Secrets\nAll hardcoded secrets have been replaced with references to Kubernetes Secrets in the YAML files.\nNever commit actual secrets to version control.\n\n### 2. Create Kubernetes Secrets\n\n#### Method 1: Using kubectl (Development)\n\n`bash\n# Create generic app secrets\nkubectl create secret generic app-secrets -n withops \\\n  --from-literal=SUPABASE_ANON_KEY='YOUR_SUPABASE_ANON_KEY' \\\n  --from-literal=SUPABASE_SERVICE_ROLE_KEY='YOUR_SUPABASE_SERVICE_ROLE_KEY' \\\n  --from-literal=SUPABASE_DATABASE_URL='postgresql://user:pass@host:5432/db' \\\n  --from-literal=DATABASE_URL='postgresql://user:pass@host:5432/db' \\\n  --from-literal=DATABASE_ENCRYPTION_KEY='YOUR_BASE64_KEY'\n\n# Create AI service secrets\nkubectl create secret generic ai-service-secrets -n withops \\\n  --from-literal=ANTHROPIC_API_KEY='sk-ant-YOUR_KEY'\n\n# Create GitHub service secrets (optional if not using inline secret)\nkubectl create secret generic github-service-secrets -n withops \\\n  --from-literal=GITHUB_OAUTH_CLIENT_ID='YOUR_CLIENT_ID' \\\n  --from-literal=GITHUB_OAUTH_CLIENT_SECRET='YOUR_CLIENT_SECRET' \\\n  --from-literal=GITHUB_APP_ID='YOUR_APP_ID' \\\n  --from-literal=GITHUB_APP_CLIENT_ID='YOUR_APP_CLIENT_ID' \\\n  --from-literal=GITHUB_APP_CLIENT_SECRET='YOUR_APP_CLIENT_SECRET' \\\n  --from-literal=GITHUB_WEBHOOK_SECRET='YOUR_WEBHOOK_SECRET' \\\n  --from-literal=GITHUB_TOKEN='ghp_YOUR_TOKEN' \\\n  --from-literal=AUTH0_CLIENT_ID='YOUR_AUTH0_ID' \\\n  --from-literal=AUTH0_CLIENT_SECRET='YOUR_AUTH0_SECRET' \\\n  --from-literal=JWT_SECRET_KEY='YOUR_JWT_SECRET'\n`\n\n#### Method 2: Using Secret Files (Recommended for Production)\n\n1. Create a secure secret file:\n`bash\ncat > secrets.env << 'EOF'\nSUPABASE_ANON_KEY=YOUR_KEY\nSUPABASE_SERVICE_ROLE_KEY=YOUR_KEY\nSUPABASE_DATABASE_URL=postgresql://user:pass@host:5432/db\nDATABASE_URL=postgresql://user:pass@host:5432/db\nDATABASE_ENCRYPTION_KEY=YOUR_KEY\nANTHROPIC_API_KEY=sk-ant-YOUR_KEY\nEOF\n\nchmod 600 secrets.env\n`\n\n2. Create secret from file:\n`bash\nkubectl create secret generic app-secrets -n withops \\\n  --from-env-file=secrets.env\n\n# Remove the file after creating secret\nrm secrets.env\n`\n\n#### Method 3: Using Sealed Secrets (Production Best Practice)\n\nSealed Secrets encrypts secrets at rest in etcd.\n\n`bash\n# Install Sealed Secrets controller\nkubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.24.0/controller.yaml\n\n# Create a sealed secret\necho -n 'YOUR_SUPABASE_ANON_KEY' | kubectl create secret generic app-secrets \\\n  -n withops \\\n  --dry-run=client \\\n  --from-file=SUPABASE_ANON_KEY=/dev/stdin \\\n  -o yaml | kubeseal --format yaml > app-secrets-sealed.yaml\n\n# Apply sealed secret (auto-decrypts in cluster)\nkubectl apply -f app-secrets-sealed.yaml\n`\n\n#### Method 4: Using External Secrets Operator (Recommended for Production)\n\nIntegrate with AWS Secrets Manager, HashiCorp Vault, or Azure Key Vault:\n\n`yaml\napiVersion: external-secrets.io/v1beta1\nkind: SecretStore\nmetadata:\n  name: aws-secretstore\n  namespace: withops\nspec:\n  provider:\n    aws:\n      service: SecretsManager\n      region: us-east-1\n      auth:\n        jwt:\n          serviceAccountRef:\n            name: external-secrets-sa\n---\napiVersion: external-secrets.io/v1beta1\nkind: ExternalSecret\nmetadata:\n  name: app-secrets\n  namespace: withops\nspec:\n  refreshInterval: 1h\n  secretStoreRef:\n    name: aws-secretstore\n    kind: SecretStore\n  target:\n    name: app-secrets\n    creationPolicy: Owner\n  data:\n    - secretKey: SUPABASE_ANON_KEY\n      remoteRef:\n        key: withops/supabase/anon-key\n    - secretKey: SUPABASE_SERVICE_ROLE_KEY\n      remoteRef:\n        key: withops/supabase/service-role-key\n    # ... add more secrets\n`\n\n### 3. Generate Secure Keys\n\n`bash\n# Generate DATABASE_ENCRYPTION_KEY (32 bytes, base64 encoded)\nopenssl rand -base64 32\n\n# Generate JWT_SECRET_KEY (minimum 32 characters)\npython3 -c \"import secrets; print(secrets.token_urlsafe(32))\"\n\n# Generate webhook secret\npython3 -c \"import secrets; print(secrets.token_hex(32))\"\n`\n\n### 4. Verify Secrets Created\n\n`bash\n# List secrets in withops namespace\nkubectl get secrets -n withops\n\n# View secret keys (not values)\nkubectl get secret app-secrets -n withops -o json | jq '.data | keys'\n\n# Test secret availability in a pod\nkubectl run -it --rm debug --image=busybox --restart=Never -n withops -- env | grep DATABASE_URL\n`\n\n### 5. Deploy Application\n\n`bash\n# Make sure secrets are created first\nkubectl get secret app-secrets -n withops\n\n# Deploy\nkubectl apply -f k8s/\n`\n\n## Security Best Practices\n\n1. **Never commit secrets to Git**\n - Use `.gitignore` to exclude `secrets.env`, `.env` files\n - Use `git-secrets` hook to prevent accidental commits\n\n2. **Use RBAC to limit secret access**\n `yaml\n   apiVersion: rbac.authorization.k8s.io/v1\n   kind: Role\n   metadata:\n     name: secret-reader\n     namespace: withops\n   rules:\n   - apiGroups: [\"\"]\n     resources: [\"secrets\"]\n     verbs: [\"get\", \"list\"]\n     resourceNames: [\"app-secrets\", \"ai-service-secrets\"]\n   `\n\n3. **Rotate secrets regularly**\n - Update secrets every 90 days\n - Use automatic rotation with External Secrets Operator\n\n4. **Encrypt secrets at rest**\n - Enable etcd encryption in Kubernetes\n - Use Sealed Secrets or External Secrets\n\n5. **Audit secret access**\n - Enable Kubernetes audit logging\n - Monitor secret access with kube-apiserver audit logs\n\n6. **Use separate secrets for different environments**\n - `app-secrets-dev`\n - `app-secrets-staging`\n - `app-secrets-prod`\n\n7. **Implement pod security policies**\n - Prevent privileged pods from reading secrets\n - Use Pod Security Standards\n\n## Environment-Specific Secrets\n\n### Development\n`bash\nkubectl create secret generic app-secrets -n withops \\\n  --from-literal=SUPABASE_ANON_KEY='dev_key' \\\n  --from-literal=DATABASE_URL='postgresql://dev:dev@localhost:5432/withops'\n`\n\n### Staging\n`bash\nkubectl create secret generic app-secrets -n withops-staging \\\n  --from-literal=SUPABASE_ANON_KEY='staging_key' \\\n  --from-literal=DATABASE_URL='postgresql://staging:pass@staging-db:5432/withops'\n`\n\n### Production\nUse Sealed Secrets or External Secrets for production deployments.\n\n## Troubleshooting\n\n### Secret not found error\n`bash\n# Check if secret exists\nkubectl get secret app-secrets -n withops\n\n# Recreate if missing\nkubectl create secret generic app-secrets -n withops --from-env-file=secrets.env\n`\n\n### Pod not reading secrets\n`bash\n# Check pod environment\nkubectl exec -it <pod-name> -n withops -- env | grep SUPABASE\n\n# Check pod logs for errors\nkubectl logs -n withops <pod-name>\n`\n\n### Secret value is incorrect\n`bash\n# Delete and recreate\nkubectl delete secret app-secrets -n withops\nkubectl create secret generic app-secrets -n withops --from-env-file=secrets.env\n\n# Roll restart deployment to pick up new secret\nkubectl rollout restart deployment/backend -n withops\n`\n\n## References\n\n- [Kubernetes Secrets Documentation](https://kubernetes.io/docs/concepts/configuration/secret/)\n- [Sealed Secrets](https://github.com/bitnami-labs/sealed-secrets)\n- [External Secrets Operator](https://external-secrets.io/)\n- [kube-apiserver Audit Logging](https://kubernetes.io/docs/tasks/debug-application-cluster/audit/)\n"
